论文阅读笔记 - Scaling Memcache at Facebook

Memcached 是一个知名的，简单的，全内存的缓存方案。这篇文章描述了facebook是如何使用memcached来构建和扩展一个分布式的key-value存储来为世界上最大的社交网站服务的。

Facebook作为一个提供SNS的网站，由于业务的需求其基础架构需要满足以下几点：

  1. 近乎实时的交流
  2. 即时聚合不同来源的内容
  3. 访问和更新非常热门的共享内容
  4. 每秒处理几百万的用户请求

Overview

我们可以从它要提供的服务以及个人的使用经验判断出这是一个Read-heavy型的服务，即在IO中读操作是主要的操作。那么自然而然的，为了提高读操作的效率和响应时间，增加缓存是我们常用的优化方式。

本文关注于memcached，这是一个全内存哈希表的开源实现，它以较低的开销提供了对共享存储的低迟延访问。另外一种内存哈希实现是Redis，都常常用来当做大型服务的缓存。

Query Cache

对于查询缓存，我们使用memcache来做为旁路缓存系统(demand-filled look-aside),如果对操作系统有所了解的话其实很熟悉这种机制。

当一个读请求来的时候，我们首先在缓存中查看是否有我们需要的数据(也成为Hit),如果没有hit，那么我们再去数据库中寻找。

在写操作的时候，我们直接往数据库中进行写操作，同时因为数据发生了改变，原来的缓存过期了，我们通知缓存删除。

在应对MySQL数据库繁重的查询通信的众多方法中，我们选择了memcache，在有限的资源与时间限制下，这是最好的选择。此外，缓存层与持久层分离，让我们可以在工作负载发生变化时快速地调整。

Generic cache

我们同样让memcache成为一个更加通用的键-值储存系统。比如说，工程师们使用memcache保存复杂的机器学习算法的中间结果，这些结果能被很多其它应用程序所使用。它只需要我们付出很少的努力，就可以让新增的服务利用现有的正在使用的基础设施，而无需调整、优化、调配和维护大型的服务器群。

扩展：从单机到分布式

Facebook使用的memcached开源实现是基于单机的，它仅仅是运行在单机上的一个内存哈希表。下文中介绍了怎样基于memcached构建一个分布式键值储存系统，以胜任在Facebook的工作负载下的操作。

上图是最终的架构，将并置集群组织起来，形成一个群体（region），指定一个主群体（master），由主群体提供数据流让非主群体保持数据同步。

In a Cluster: Latency and Load

如上面所说的一样，当我们以集群的形式面对上千台服器时，我们希望提高缓存的命中率，从而减少数据库的负载。

Reducing Latency

用于一个页面需要加载的内容非常多，当用户发送一个web request的时候，可能会触发数百次的memcache的get request。当同时面对如此大量的request请求的时候，负载均衡是一个很容易想到的解决办法，就如同Dynamo一样，可以采用一致性哈希将数据分布到不同的memcached服务器上，这样当request来的时候，一个web server就要与许多个不同的memcached server通信来满足一个user request。这样做的结果是，许多web servers要在短时间内与许多memcached servers通信，有点类似于洪泛的状况发生，会导致通信的阻塞，这种情况学术上叫做incast congestion。

要解决这个问题，我们把优化的重心放在memcache客户端，它运行在每一个web server上。每一个客户端有一张map,来保存哪些memcached server是可用的。

Clients maintain a map of all available servers, which is updated through an auxiliary configu- ration system.

Parallel requests and batching

我们通过研究这个通信网络的结构来减少通信过程中的不必要的来回通信。通过建立一个有向无环图来代表用户所需要的数据之间的依赖，我们尽最大可能来避免向不同server提交无意义的请求。

Client-server communication

第二点是，我们只允许client server与memcached server通信，client server之间的通信。但是我们不允许memcached server之间通信。这样做的原因是我们需要保证memcached server是无状态的，这样可以保证我们可以很方便的进行扩展。

同时，我们使用一个叫做mcrouter的代理来分发request请求。

This proxy presents a memcached server interface and routes the requests/replies to/from other servers.

Client同时使用了UDP和TCP协议来与memcached server进行通信。

在调用get request的时候，我们使用UDP协议。由于UDP协议是无连接的，因此通信开销较小，虽然它不保证消息一定送达，但是由于我们有replica，只要收到一次消息就可以了，这样的准确性还是非常可靠的。 而在set和delete request的时候，由于我们需要保证操作的成功，必须使用TCP协议。

Incast congestion

为了解决上文中提到过的通信网络不堪重负的情况，也就是Incast congestion，我们采用滑动窗口(sliding window mechanism)的方法来解决此问题。有点类似于一个消息队列，我一次只能处理这么多请求，多余的就给我从后面等着。这个窗口的大小是动态分配的，如果我上一次的request处理成功了，窗口大小就缓慢增加，否则就缓慢减小。

Reducing Load

我们使用memcached的目的就是想通过缓存尽量减少request直接读写数据库的压力，不然很容易扛不住，但是当缓存没有命中的时候，request终究还是要走数据库的。因此我们要想办法避免这种情况发生的可能性。

Handling Failures

在memcache server集群中，容错性也是必不可少的，其中有两类failure我们必须要注意：

  1. 由于网络等原因，一小部分主机无法连接了。
  2. 范围较广的outage对集群中的相当一部分机器造成了影响。

对于一小部分主机无法连接，我们通常使用一个自动恢复系统就可以。这个系统恢复的时间需要几分钟，看似不长，实际可能会对整个服务造成很严重的影响。

为了解决这个问题，我们单独拿出memcached集群中的一小部分机器(1%)，当做救火部队，叫做Gutter。当客户端的get request一段时间内得不到memcache服务器回应时，我们就默认这台服务挂了，这时客户端会像Gutter服务器发出get请求。如果得到反馈最好，如果还是hit miss的话，我们没办法只能走最原始的数据库查找了，但是这里要注意的是，当我们从数据库中找到结果后，顺便还要把Gutter服务器给更新了，确保下次不用再走数据库。

这里我们再发散一下，为什么我们不选择另一种做法呢：rehash一下把keys在重新分配到没有挂的所有服务上去器呢？这是由于在实践中，访问key的频率并不是平均分布的。举个例子，也许某种key的访问请求占用了所有访问请求的50%,那么即使通过rehash的方式重新分配了服务器，负责该key的服务器仍然容易overload挂掉。

Gutter的作用其实就是当正常的memcache服务过载或者挂掉的时候，我在直接读取数据库之前再加一层缓冲，通过这层缓冲来缓解直接对数据库读写的压力，效果拔群哦~

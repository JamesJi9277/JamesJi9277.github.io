[Java网络编程]2. Stream

Java中的I/O是通过stream的形式实现的，普通的包括InputStream和OutputStream, 比较特殊的还有FilterStream.

Stream是线程同步的，通俗的说就是

when a program (really a thread) asks a stream to read or write a piece of data, it waits for the data to be read or written before it does anything else.

当然Java同样也提供nonblocking I/O，是通过channels和buffers实现的。

尽管write(int b)传入的参数为int型，它世界上写入的是一个unsigned byte.

Java doesn’t have an unsigned byte data type, so an int has to be used here instead.

每一次只输出一个byte无疑是低效的，这在TCP协议中尤其明显，因为这样报头等信息占据了传输的大部分内容。一种很明显解决办法是缓存起来一起发送。write(byte[] data)和write(byte[] data, int offset, int length)就是用来做这个事情的。缓存的方法可以使用BufferedOutput Stream或者BufferedWriter。

使用buffer可能会出现一个问题，即如果只有buffer满的时候才会发送数据的话，在buffer不满的情况下，我们的request发不出去，因而无法得到response，这样可能会导致死锁的发生。flush()方法就是为了解决这一问题而实现的。

The flush() method breaks the deadlock by forcing the buffered stream to send its data even if the buffer isn’t yet full.

As a corollary to all this, you should flush all streams immediately before you close them. Otherwise, data left in the buffer when the stream is closed may get lost.

最后，当我们使用完毕stream后，可以调用close()方法关闭stream. 如果Stream不能正常关闭的话可能会导致一系列问题，比如端口和文件的持续占用等等，因此很多习惯于使用finally block来关闭stream.

这样做除了比较繁琐以外，有两个需要注意的地方：

  1. outputstream的声明要在try语句以外，提前声明为null
  2. 在finally中关闭stream的时候，要check一下stream是否为null，如果为Null就不用关闭了。

假设我的buffer长度为1024，在网络通信中可能由于各种原因当前时刻我只收到了500的数据，此时read不会等待所有数据都到达才开始读取，而是buffer里面有多少读多少，直到buffer为空为止。

当stream中所有的数据都到达且读取完毕时，我们再读取buffer中的内容，此时会返回-1，这时才是应当调用close关闭stream的时候。那么我想只读取里面有的data但是不关闭stream应该怎么办呢，我又不想让他返回-1.这就是available()函数的作用了，该函数可以返回我们仍然需要读取的data的数量，当这个函数返回0时，我们关闭stream即可。

有时候我们希望跳过一些data不读，这时可以调用skip()函数，在读取文件的时候这个函数比较好实现，因为文件在OS中是random access的，我们只需要改一下pointer即可，但是在网络中数据都是连续的。

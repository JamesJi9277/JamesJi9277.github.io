论文阅读笔记 - Dynamo: Amazon’s Highly Available Key-value Store

What is Dynamo?

Dynamo是Amazon设计的一种key-value storage system，不同于Amazon的S3(Simple Storage Service), Dynamo的设计初衷就是提供高可靠性、高可扩展性的数据存储服务，为此甚至不惜牺牲在某些特定failure条件下的一致性。

我觉得下面摘录的论文中的一段话可以很好的总结Dynamo的特点，如果对于其中的某些术语不甚了解，那么你应当start from bottom, 首先对分布式系统中的基础知识有所了解。

Dynamo uses a synthesis of well known techniques to achieve scalability and availability: Data is partitioned and replicated using consistent hashing, and consistency is facilitated by object versioning.

The consistency among replicas during updates is maintained by a quorum-like technique and a decentralized replica synchronization protocol.

Dynamo employs a gossip based distributed failure detection and membership protocol.

系统架构设计

由于篇幅原因，论文中主要介绍了Dynamo系统设计的以下几个方面：

  • partitioning
  • replication
  • versioning
  • membership
  • failure handling and scaling

System Interface

Dynamo的系统接口设计的跟简单，跟dict一样使用put()和get()方法即可，稍微有点区别的地方在于方法中需要带上存取的object的context信息。

The context encodes system metadata about the object that is opaque to the caller and includes information such as the version of the object.

Dynamo通过MD5 hash对key进行哈希，生成一个128位的identifier来决定每个key所对应的位置。

Partitioning Algorithm

Dynamo的partitioning主要依靠一致性哈希来将不同的key映射的到不同的位置。

In consistent hashing, the output range of a hash function is treated as a fixed circular space or “ring”

通过一致性哈希，key被映射到一个环状结构的不同位置，并且按顺时针方向存储到下一个位置上。

Thus, each node becomes responsible for the region in the ring between it and its predecessor node on the ring.

通过一致性hash,一个节点的增删改只影响相邻的节点，而不会对其他节点造成影响。

Dynamo对一致性哈希做了改进，不是将一个节点单纯的映射到环状区域的一个位置，而是引入了一个”Virtual Node”的概念，这样，一个Node会被映射到多个不同的位置。

A virtual node looks like a single node in the system, but each node can be responsible for more than one virtual node.

Replication

Dynamo在做Replication的时候，每个数据都会备份在N个host上，通过hash我们首先将一个key映射到某个位置，这个位置叫做coordinator，coordinator除了在本地备份数据外，还会在顺时针方向下面的N-1个host上也同时备份数据，就跟上一节中介绍的的一致性哈希一样。

同时，我们引入一个叫做preference list的概念:

The list of nodes that is responsible for storing a particular key is called the preference list.

Data Versioning

Dynamo提供的是eventual consistency，也就是说，对于replica的更新是异步的，并不保证当我更新数据后所有replica可以同时更新。这样的异步设计降低了用户等待反馈的时间，但是在这个过程中，如果在有限时间内，由于网络等原因无法完成所有replica的更新，会造成一个问题：

在同一数据的所有replica中，存储的数据的版本是不同的，如果此时我调用get()方法来得到数据的话，会得到不是唯一的结果。

[Java网络编程]3. Threads

多线程是网络编程中常用的方法，轻量级的线程比进程更加适合并发操作，但是线程之间共享空间也会导致我们在线程同步的过程中非常的小心谨慎才行。

在I/O密集型应用中，如果我们每个县城对应一次连接，对于每一次短连接我们都要开关线程无疑是一件很浪费的事情，常用的做法是维护一个线程池，把每个线程当做一个worker，只有worker空闲的时候才会去接受新的任务，堆积的任务保存在buffer队列中，这种思想也很常见就不多说了。

当然如果一个应用需要非常多的长连接，那么异步I/0显然是一种更好的方案，我们将在以后讨论这个话题。

Returning Information from a Thread

很多时候我们需要得到一个thread运行完之后的返回值，常见的做法是我在类中用一个变量存储结果，然后通过一个getter拿出来。这样的问题在于，在多线程中并不保证顺序，如果我们在result没有产生之前调用getter并不能拿到我们想要的结果。

Race Conditions

Whether you get the correct results, an exception, or a hung program depends on many factors, including how many threads the program spawns, the speed of the CPU and disk on the system where this is run, how many CPUs the system uses, and the algorithm the Java virtual machine uses to allot time to different threads. This is called a race condition.

Race Condition是在OS中遇到了很多遍了，根本原因在于各个线程的进度是不可控的无序的，你很难保证结果的可复现性，有时候运气好跑得出来结果，有时候就跪了。

Polling

一种被很多人采用的解决race condition的方法是polling，即我设置一个标志位来记录result是否已经产生了。然后在主线程中周期性的探测这个flag的值，来判断我们是否可以通过getter来得到result了。

这种方法尤其可取之处，但是缺点也显而易见，那就是主线程很可能一直在不断的check，占有了CPU的大量时间，使得应用变得极其缓慢。
